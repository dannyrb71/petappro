How to use this playbook

This is Part 1 — Foundations. Each block below is a self-contained prompt you paste into ChatGPT to generate a piece of the Figma file, in order. The prompts define rules, not one-off objects, so what comes back scales.

Recommended workflow. Install the Tokens Studio for Figma plugin first — several prompts ask ChatGPT to output token JSON you import directly (variables + theme modes land automatically). Prompts that build frames/components give step-by-step Figma instructions instead. Run them top to bottom; later prompts assume earlier variables exist.

Every prompt already contains our real values (hex, scale, type). Where ChatGPT needs judgment, the prompt tells it the rule to follow. After each result, our “design system lead” pass checks spacing consistency, variant necessity, missing states, accessibility, and Auto Layout before it joins the library. Dark mode is intentionally out of scope for now — the token structure supports adding it later as an extra mode without rework.

The “How this file works” statement

This is finished copy — place it as a text block on the Cover (or its own “Read me” frame). No prompt needed; paste it directly.

CopyHow this file works
HOW THIS FILE WORKS

This is the single source of truth for the Petappro platform UI. Design with tokens, not raw values.

TOKEN TIERS
1. Primitives — raw palette ramps, type sizes, and the spacing scale. Never applied to a layer directly.
2. Theme — semantic roles (primary, surface, on-surface, outline, status…). These change per brand via MODES. This is the layer you actually apply.
3. Component — per-component hooks (button shape, card radius, input container) that map onto Theme roles, so a single edit re-tunes every instance.

THEMING (WHITE-LABEL)
The "Theme" variable collection has one mode per theme: Sage & Sand (default), Terracotta, Harbor, Dusk, Berry. Switch a frame's mode to re-skin it — never hardcode a hex. Middle-tier providers pick one mode. Top-tier white-label providers get a new custom mode (+ optional font family from the approved set).

PAGES
Cover · Foundations · Atoms · Molecules · Organisms · Templates.

RULES
- Apply Theme/Component variables to layers; never raw primitives or literal hex.
- All components use Auto Layout and named component properties.
- Every interactive component ships all states: default, hover, pressed, focus-visible, disabled, loading (where relevant).
- Meet WCAG 2.1 AA: body text ≥ 4.5:1, large/UI ≥ 3:1, touch targets ≥ 44px.
- Property + variant names are chosen to map cleanly onto the eventual React Native implementation.
- Dark mode is not built yet; the token structure supports adding it later as an extra mode.
02 · Foundations

Variable architecture & naming

Establishes the three collections and the naming convention every later prompt relies on. Run this before any other foundation prompt.

PromptSet up variable collections & naming rules
Set up the Figma Variables architecture for the Petappro design system. Output as Tokens Studio for Figma-compatible JSON (token sets), plus a short written summary of the naming convention.

Create THREE token sets (these map to Figma variable collections):

1. "primitives" — raw values, no modes. Naming: category/scale.
   Examples: color/pine/700, color/sage/500, color/sand/100, size/font/15, space/4, radius/4.

2. "theme" — semantic roles that reference primitives. This set is MULTI-MODE (one mode per brand theme). Naming: role.subrole.state.
   Groups: color.primary(.hover/.pressed/.container/.on/.on-container), color.secondary(...), color.surface(.container/.container-high), color.on-surface(.body/.variant/.accent), color.outline(.variant/.strong), color.focus-ring, plus status roles (see the business-logic prompt later).

3. "component" — per-component hooks referencing theme roles. Naming: comp...
   Examples: comp.button.shape, comp.button.height, comp.card.shape, comp.card.elevation, comp.input.container.

Rules:
- Semantic first: components must reference "theme"/"component" tokens, never "primitives" directly.
- Every color token stores a reference (alias) to a primitive, not a literal — except inside "primitives".
- Token names use lowercase, dot/slash separators, and read as role → subrole → state so they translate directly to React Native style keys later.
- Do not invent tokens beyond what I supply in later prompts; this prompt only creates the empty structure + naming rules + a couple of example tokens per set.

Return: the JSON, then a 6-line naming cheat-sheet.

06 · Spacing scale

A 4px base grid. Applied through Auto Layout gaps and padding — never ad-hoc numbers.

PromptSpacing variables + usage rules
Confirm the "space" primitives (4px grid): 0,4,8,12,16,20,24,32,40,48,64 as space/0..16.

Then write the SPACING RULES for the design system (as a text block for the Foundations page):
- All padding, gaps, and margins use a space token; no arbitrary pixel values.
- Auto Layout gap and padding must reference space tokens.
- Default card padding = 20 (space/5). Default section gap = 20. Tight groups = 8/12.
- Touch targets ≥ 44px regardless of spacing.
Return the confirmed token list + the rules text.

What's next: Atoms

Once the foundation variables + styles exist, Part 2 builds the atoms as specifications (Objective → Requirements → Acceptance Criteria), gated by the Governance charter. Here's the Button spec:

SpecCanonical Button component
OBJECTIVE
Create the canonical Button for the Petappro Design System.

REQUIREMENTS
- Semantic variants: Primary, Secondary, Tertiary/Ghost, Destructive, and Success (booking-green — used for booking-CREATION actions only).
- Sizes: Lg 52 / Md 44 / Sm 36 / Xs 28 (px height); Md is the default and the standard tap target.
- Icon slots: optional leading + trailing (Lucide, 1.9 stroke), sized to match the label.
- States: default, hover, pressed, focus-visible (3px ring, ≥3:1), disabled, loading (spinner replaces label, button width locked).
- Tokens: fill/label from theme color roles; corner radius bound to comp.button.shape (fully rounded); gap = space/2; height per size token.
- Auto Layout: horizontal, hug contents, center-aligned, padding from space tokens.
- React Native compatibility: expose component properties (variant, size, leadingIcon, trailingIcon, state, label) named to map onto the RN prop names.

ACCEPTANCE CRITERIA
- No hard-coded colors or literal hex; all variants use shared tokens.
- No detached instances.
- Meets WCAG AA (contrast + focus ring + 44px default target).
- Minimal, well-named component properties.
- One reusable component — no page-specific copies.

DOCUMENTATION
- Purpose · Usage · When not to use · Variant descriptions (incl. "booking-creation = Success green") · A11y notes · Dev notes.

Then the rest of the atoms (Input, Avatar, Icon, Money, Status Badge, Service Pill), molecules (Section Header, Date Navigator, Dog Card, field groups, pricing row), organisms (Pricing table, Reservation card, Dashboard data tiles, Household card), and templates (Dashboard, Booking, Profile, Schedule). We'll also add a Business-logic foundation block — status, booking, payment, notification, and role colors — pulled from your live app.
